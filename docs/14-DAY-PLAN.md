# PitchLogic — 14 天冲刺计划

**日期:** 2026-04-08 → 2026-04-21
**目标:** 让分析管道真正可用 — 精准、快速、能跑长视频、能出 AI 总结

---

## 五大核心目标

| # | 目标 | 为什么重要 |
|---|------|-----------|
| 1 | **精确检测** — 小地图精准、球检测不丢、控球率准确、控球球员识别 | 数据不准等于没用 |
| 2 | **速度提升** — 检测+追踪管道至少快 3-5x | 等 30 分钟出结果没人用 |
| 3 | **长视频支持** — 稳定处理 20min~90min 比赛视频 | 真实比赛都是长视频 |
| 4 | **中场休息检测** — 自动识别上下半场，分段分析 | 中场混乱帧会污染数据 |
| 5 | **多模态 AI 总结** — 接入视觉大模型生成战术分析报告 | 从"看数据"升级到"看懂比赛" |

## Phase 1: 精确检测 (Day 1-5)

> 让每一个数据点都可信

### Day 1-2 (Apr 8-9) — 球检测 + 控球球员

**现状问题:**
- 球检测 conf=0.1 太低，误检多
- `SmartBallPossessionDetector` 用固定像素阈值（70px），不同分辨率下不一致
- 控球球员判定仅靠"最近距离"，没有考虑球运动方向

**要做的:**
- [ ] 球检测专用模型：用足球专用 YOLO 权重，或调高 ball confidence 到 0.3+
- [ ] 球轨迹插值优化：当前 `interpolate_ball_positions()` 是线性插值，改成抛物线/样条插值更符合物理
- [ ] 控球球员检测重写：
  - 距离阈值改为相对值（bbox 高度的 1.5 倍）而非固定 70px
  - 加入球运动方向判断：球朝某球员飞 → 该球员即将控球
  - 加入脚部位置估算（bbox 底部中心）而非 bbox 中心
- [ ] 控球状态添加第四种："loose ball"（无人控球，球在地面缓慢移动）
- [ ] 写单元测试验证控球检测精度

### Day 3-4 (Apr 10-11) — 小地图精度

**现状问题:**
- `KeypointDetector` stride=60（每 60 帧才检测一次关键点），中间帧用最近邻填充，会跳变
- 透视变换需要至少 4 个关键点，不足时直接跳过，导致小地图断断续续
- Gaussian 平滑 window=25 太大，快速移动时位置延迟

**要做的:**
- [ ] 关键点检测 stride 降到 15-20，用线性插值（而非最近邻）填充中间帧
- [ ] 关键点不足 4 个时：用上一帧的 homography matrix 做 fallback
- [ ] 平滑窗口自适应：静止时 window=15，移动时 window=5
- [ ] 添加球场边界约束：球员坐标不能超出球场范围（clamp 到 0-105m × 0-68m）
- [ ] 小地图渲染加入球的轨迹线（最近 30 帧）
- [ ] 对比测试：old vs new 小地图，录屏对比

### Day 5 (Apr 12) — 控球率 + 队伍分配精度

**现状问题:**
- 队伍颜色用 KMeans(k=2) 只取上半身，球衣颜色相近时分不开
- 控球率只算"哪个队的球员离球最近"，不考虑控球时长权重

**要做的:**
- [ ] 队伍分配升级：
  - HSV 颜色空间替代 RGB（对光照变化更鲁棒）
  - 采样区域：去掉皮肤色区域（只取球衣核心区域）
  - 增加投票帧数：8 → 16 帧
  - 添加置信度分数：低置信度的分配标记为 uncertain
- [ ] 控球率计算升级：
  - 基于"实际控球时间"而非"每帧最近距离"
  - 只有 state=="controlled" 时才计入控球率
  - 添加"控球切换次数"统计
- [ ] 集成测试：用已知比赛片段验证控球率 vs 官方数据

## Phase 2: 速度 + 长视频 (Day 6-9)

> 20 分钟视频也能在合理时间内跑完

### Day 6-7 (Apr 13-14) — 管道速度优化

**当前瓶颈分析:**
```
SAMURAI 追踪:  resize 0.5x + stride=10 → 已优化
YOLO 检测:    每帧检测, batch=60, imgsz=416 → 主要瓶颈
光学流:       stride=3 → 可接受
关键点检测:    stride=60 → 可接受
小地图渲染:    每帧 matplotlib → 极慢
```

**要做的:**
- [ ] YOLO 检测优化：
  - 非关键帧用 stride=2-3（每 2-3 帧检测一次），ByteTrack 负责插值
  - 开启 TensorRT/CoreML 加速（Mac 用 CoreML）
  - imgsz 416 → 320（对球员检测影响小，速度提升 ~40%）
  - 用 `model.track()` 替代手动 `predict()` + ByteTrack（YOLO 内建 tracking 更快）
- [ ] 小地图渲染优化：
  - `render_minimap_frame()` 去掉 matplotlib，改用纯 OpenCV 绘制（10x+提速）
  - 预渲染球场背景，每帧只画球员点
- [ ] 并行化：
  - YOLO 检测 + 光学流可以并行（不同线程处理不同帧段）
  - 特征生成（heatmap/speed_chart/possession）并行触发
- [ ] 添加 benchmark 脚本：输入视频 → 输出各阶段耗时

### Day 8-9 (Apr 15-16) — 长视频支持

**现状问题:**
- `read_video()` 把所有帧加载到内存 → 20min 视频 ~36000帧 × 3MB ≈ 108GB RAM
- 已有 `stream_video_chunks()` 但未完全集成
- pickle 缓存文件（tracks.pkl）对长视频可能超过 1GB
- 进度条在长视频下不准确

**要做的:**
- [ ] 全流程流式化验证：确保 0 个函数还在用 `read_video()`
- [ ] tracks 数据分段存储：
  - 每 3000 帧存一个 chunk（`tracks_chunk_0.pkl`, `tracks_chunk_1.pkl`...）
  - 加载时按需读取，不全部加载
- [ ] 进度报告精细化：
  - YOLO 阶段：每 500 帧报告一次进度（帧数/总帧数）
  - 显示 ETA（估算剩余时间）
- [ ] 内存监控：添加 `psutil` 内存使用日志，超过阈值时 GC
- [ ] 测试：用 20 分钟视频端到端跑通，记录内存峰值和总耗时
- [ ] 超长视频方案：90 分钟视频分成 15 分钟 segment 处理，最后合并结果

## Phase 3: 中场检测 + AI 总结 (Day 10-14)

> 智能识别比赛结构，生成人类可读的分析报告

### Day 10-11 (Apr 17-18) — 中场休息 / 场景切换检测

**核心思路:**
比赛视频中，中场休息的特征：
1. 球员突然从球场消失（检测数量骤降）
2. 镜头切换到观众/回放/广告
3. 持续 10-15 分钟的非比赛画面

**要做的:**
- [ ] 场景变化检测器 `SceneChangeDetector`：
  - 方法 1：统计每帧检测到的球员数，连续 N 帧球员数 < 5 → 标记为"非比赛段"
  - 方法 2：帧间像素差异（`cv2.absdiff`），突变 = 场景切换
  - 方法 3：颜色直方图比较，绿色（草地）占比骤降 = 离开球场
- [ ] 比赛分段：
  ```
  输入: 90min 视频
  输出: [
    {"type": "first_half",  "start": 0,     "end": 47min},
    {"type": "halftime",    "start": 47min, "end": 62min},
    {"type": "second_half", "start": 62min, "end": 110min}
  ]
  ```
- [ ] 只对比赛段运行完整分析管道，非比赛段跳过
- [ ] 上下半场独立统计 + 合计统计
- [ ] 前端 Dashboard 添加半场切换 tab

### Day 12-13 (Apr 19-20) — 多模态 AI 总结

**现状:** `AIInsights.jsx` 是硬编码的 mock，没有真正的 LLM 调用

**架构设计:**
```
分析数据 (JSON)                   关键帧截图 (PNG)
    ↓                                  ↓
    └──────── Claude API ──────────────┘
              (claude-sonnet-4-6)
                    ↓
            结构化战术分析报告
```

**要做的:**
- [ ] 后端新增 `ai_summary` 任务：
  - 收集输入数据：
    - 控球率、控球切换次数
    - 各球员速度/距离/热图数据
    - 关键时刻截图（控球切换、高速冲刺）
    - 小地图截图（阵型快照）
  - 调用 Claude API (sonnet)：
    - System prompt: "你是一个专业足球战术分析师..."
    - 多模态输入：JSON 数据 + 3-5 张关键帧图片
    - 输出格式：markdown 结构化报告
- [ ] 报告内容模板：
  ```markdown
  ## 比赛概览
  - 控球率对比、跑动距离对比

  ## 战术分析
  - 阵型变化（基于小地图位置数据）
  - 进攻模式（基于控球切换位置）
  - 防守强度（基于球员密度热图）

  ## 球员表现
  - 跑动最多 / 最快 / 控球最多的球员
  - 关键位置活动区域

  ## AI 观察
  - 比赛节奏变化
  - 值得关注的战术趋势
  ```
- [ ] 前端 `AIInsights.jsx` 改为真实调用：
  - 显示 loading 状态（"AI 正在分析..."）
  - 流式输出（打字机效果已有，对接真实 stream）
  - 缓存结果，不重复调用
- [ ] 添加"重新生成"按钮

### Day 14 (Apr 21) — 集成测试 + Buffer

- [ ] 用一段完整的 20 分钟比赛视频做端到端测试：
  - 上传 → 自动检测 → 追踪 → 分析 → 所有特征生成 → AI 总结
  - 记录总耗时、内存峰值、各阶段耗时
- [ ] 精度验证：

  - 小地图位置 vs 视频实际位置，目测合理
  - 球员速度 vs 已知数据（职业球员最高速 ~35km/h）
- [ ] 修复集成测试发现的 bug

## 技术决策速查

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 球检测模型 | 先调参，不行再换专用模型 | 换模型成本高，先试低成本方案 |
| 小地图渲染 | OpenCV 替代 matplotlib | matplotlib 太慢，不适合逐帧渲染 |
| 长视频存储 | 分段 pickle chunks | 比数据库简单，比单文件省内存 |
| 中场检测 | 球员数量统计法 | 最简单可靠，不需要额外模型 |
| AI 总结模型 | Claude Sonnet | 多模态能力强，API 稳定，性价比高 |
| YOLO 加速 | stride + CoreML | Mac 上 CoreML 是最快的推理方式 |

## 风险管理

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 球检测精度仍然不够 | 中 | 控球率不准 | 用专门训练的 ball detection 模型 |
| CoreML 转换失败 | 中 | 速度优化打折 | fallback 到 stride 优化 |
| 长视频 OOM | 高 | 崩溃 | 严格流式化 + 分段处理 |
| 中场检测误判 | 低 | 数据分段错误 | 加人工确认步骤 |
| Claude API 限流 | 低 | AI 总结失败 | 本地缓存 + 重试逻辑 |

## 成功标准

Sprint 结束时:

1. **精度**: 控球率误差 < 15%，小地图位置视觉合理，球检测不丢球
2. **速度**: 5 分钟视频分析时间 < 3 分钟（不含 SAMURAI）
3. **长视频**: 20 分钟视频跑完不 OOM，结果完整
4. **中场**: 能自动识别上下半场，分段统计
5. **AI 总结**: 上传视频后能看到 AI 生成的战术分析报告（非 mock）
